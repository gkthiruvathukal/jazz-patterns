# AGENTS.md

This file is the single source of guidance for AI coding agents (Claude Code and others) working in this repository. `CLAUDE.md` just points here.

## What this is

A monorepo of two **independent** subprojects that generate printable jazz practice material with Abjad + LilyPond, plus a shared helper package:

- `common/` → **`jazz_common`** — shared pitch-class, note-naming, and LilyPond-compile helpers.
- `projects/scales/` → **`jazz_scales`** — multi-key scale charts (forward + retrograde) and a merged book.
- `projects/blues/` → **`jazz_blues`** — annotated 12-bar blues studies (separate work; keep it decoupled from scales).
- `projects/web/` → **interactive web app** (TypeScript + Vite + VexFlow + smplr) — client-side only, renders/plays a JSON exported from the scales model.
- `projects/presentation/` → **Marp (Markdown) slide deck** about the project. Source is `slides.md` + `theme.css`; `npm run build` → static `dist/index.html`. Notation images in `assets/` are generated from the project (LilyPond engravings + web-app screenshots) and committed. Deployed to **`/slides`** on the Pages site (via `deploy-web.yml`); HTML-only (no PDF or Release artifact).

The Python subprojects depend on `jazz_common` (installed from the local path); **none of the subprojects depend on another's code**, and that boundary is intentional — do not add cross-subproject imports. Each has its own build (`build.sh` for Python, `npm`/Vite for web) and CI job. Outputs land in each subproject's own `build/` (or `dist/` for web).

The web app's data is the **JSON contract** at `projects/web/src/data/scales.json`, generated by `python -m jazz_scales.export_json` (the Python model is the single source of truth — no music theory is duplicated in TypeScript). Regenerate it whenever `SCALES` or the pitch model changes.

The pipeline (both subprojects): emit LilyPond `.ly` via Abjad → compile to `.pdf`/`.midi` with LilyPond → (scales only) render `.wav` with FluidSynth → (scales only) assemble a combined PDF book with cover + TOC.

## Commands

Each subproject's `build.sh` bootstraps a `.venv`, installs `jazz-common` from `../../common`, installs itself, and builds into its own `build/`.

Full scales build (per-key PDFs/MIDI/WAV + `Jazz-Scales-Book.pdf`):
```bash
bash projects/scales/build.sh
```

Full blues build (studies in Bb, F, C):
```bash
bash projects/blues/build.sh
```

Web app (from `projects/web`): `npm install`, then `npm run dev` (dev server),
`npm run typecheck`, or `npm run build` (static `dist/`). Regenerate its data with
`python -m jazz_scales.export_json --output ../web/src/data/scales.json` from
`projects/scales`.

For dev work, install once then run modules directly (from the subproject dir):
```bash
pip install -e ../../common && pip install -e .          # in projects/scales or projects/blues
python -m jazz_scales.generator --output-dir build --pdf --midi --bpm 96   # add --step/--count/--start/--prefer/--anchor/--mode/--no-enharmonics
python -m jazz_blues.blues_take_1 --keys Bb F C --output-dir build --pdf --midi
```
Drop `--pdf`/`--midi` to emit only `.ly` (no LilyPond needed) — useful for fast iteration and for confirming imports resolve.

Scales-only extras (run from `projects/scales`):
```bash
bash src/jazz_scales/render_wavs.sh build build                            # WAV; fetches Salamander SF2 on demand into .cache/soundfonts/
MPLCONFIGDIR="$PWD/.matplotlib" MPLBACKEND=Agg python -m jazz_scales.cover --output-dir build
python -m jazz_scales.book --output-dir build
```

There is no test suite or linter configured. Verification is by inspecting generated `.ly`/PDFs/WAVs.

## System dependencies

LilyPond (PDF/MIDI), FluidSynth (WAV, scales only). macOS: `brew install lilypond fluidsynth`. Python deps: `jazz_common`/`jazz_blues` need `abjad>=3.31`; `jazz_scales` additionally needs `matplotlib`, `pypdf`, `reportlab`.

## Architecture

### Shared: `jazz_common`
- **`pitch.py`** — keys/notes are integer pitch classes 0–11. Lookup tables (`SHARP_NAMES`/`FLAT_NAMES`, `SHARP_KEYS`/`FLAT_KEYS` for LilyPond `\language "english"` tokens, `NAME_TO_PC`, `LETTER_TO_PC`) plus `pc_to_name`, `pc_to_lily_key`, `auto_prefer_for_pc` (decides flats vs sharps **per key**, not once per batch), `numbered_pitch_from_name`, `sanitize_key_for_filename`.
- **`lilypond.py`** — `compile_with_lilypond(ly_path, want_pdf, want_midi)` shells out to the `lilypond` binary and detects success by checking for output files, not exit codes alone.

### Scales: `jazz_scales`
**`generator.py` is the core.** `book.py` consumes its output.
- **`SCALES`** — the data table that drives everything. Each entry is `(name, notes, step_pattern, chord_symbol)` written **relative to C root** (e.g. `"Cmaj7"`, notes like `["C","D","E",...]`). To add/change a scale, edit this list. The `"C5"` suffix means octave 5 (above the default 4). Step labels are hand-authored and use the vocabulary `H`/`W`/`W+H`/`m3`/`M3` = 1/2/3/3/4 semitones — **each label must equal the actual gap between consecutive notes** (a 4-semitone leap is `M3`, not `m3`). This is easy to get wrong: `Minor Pentatonic b5` (Gb→Bb is `M3`) and `Altered` (swapped `H`/`W`) were both corrected here; verify any label edit against the note pitches.
- **Two chapter organizations**, selected by `--sections {key,scale,both}` (default `both`):
  - **By key** (`build_score_for_key`): transposes the C-root `SCALES` by `pc_to_register_offset` (anchoring around middle C); per scale a forward `make_bar` (scale name + chord above, step labels below) and a `make_retrograde_bar`. Both bars now carry W/H step labels (the retrograde uses the reversed interval list). One score per key, systems split by `\break`. The set of keys comes from **`key_cycle`** (in `jazz_common.pitch`), shared with `export_json.py`: the `--start`/`--step`/`--count` pitch-class cycle (default 12 keys in fourths) **plus** the F#/C# enharmonics inserted right after Gb/Db (spelled sharp) → 14 keys; pass `--no-enharmonics` to emit just the cycle.
  - **By scale** (`build_movements_for_scale`): one scale across all keys (now 14, incl. F#/C#), each key emitted as a **separate `\score` "movement"** (via `write_lilypond_movements`) so per-key key-signature changes don't leave a trailing courtesy measure; `ragged-right = ##f` justifies each two-measure movement to full width.
- **`export_json.py`** (`python -m jazz_scales.export_json --output <path>`) emits the resolved per-(key,scale) data the web app consumes — reuses the same helpers, no notation objects.
- **Filename conventions are load-bearing**: by-key files are `jazz_scales_abjad_<key>.ly/.pdf/.midi` (`<key>` from `sanitize_key_for_filename`: `#`→`sharp`, `b`→`flat`); by-scale files are `jazz_scales_byscale_<slug>.pdf` (`<slug>` from `scale_slug`). `book.py` reverses the key mapping (`pretty_from_filename`) and uses `SCALES`/`scale_slug` for scale chapters.
- **`book.py`** merges `cover.pdf` + a generated TOC + by-key chapters then by-scale chapters into `Jazz-Scales-Book.pdf`, with a two-section TOC and outline bookmarks. TOC page numbers are resolved by iterating to a fixed point (the TOC's own page count depends on entry count), since the TOC now spans multiple pages.
- `generate_single.py` / `book_single.py` are **legacy** single-key tools kept for reference; the multi-key `generator.py` + `book.py` path is current.

### Blues: `jazz_blues`
**`blues_take_1.py`** generates annotated 12-bar blues choruses. Unlike scales, bars are authored as **literal LilyPond strings** (not the `SCALES` data model), with a `JAZZ_BLUES_FORM` chord progression and per-bar footnotes; it transposes those literals by pitch class. It imports only from `jazz_common` (pitch utils + `compile_with_lilypond`). Outputs `blues_take_1_<key>.{ly,pdf,midi}`.

### Web: `projects/web` (TypeScript)
Vanilla TS + Vite, no UI framework. Renders/plays the JSON contract `src/data/scales.json` (generated by `jazz_scales.export_json`); **no music theory is duplicated in TS** — it's a renderer/player over resolved data. Key invariant: the score is notation-only — all the "feel" controls (octave, swing, accents) change **playback audio only and never the printed notes**.
- **`data/scales.ts`** — types (`Chart`, `Note`, `ScalesData`) + `findChart`; imports the committed `scales.json`.
- **`notation.ts`** — VexFlow (v5): one treble stave, 4/4, notes padded with rests into complete measures; W/H/W+H/m3/M3 step labels (or scale-degree numbers for interval patterns) as bottom `Annotation`s; `renderChart` consumes the already-resolved play-order note + label sequence from `sequence.ts`. The SVG is made responsive by setting **`svg.style`** (width 100%, capped at natural width, with a `viewBox`) — it must be `.style`, not `setAttribute`, because VexFlow's own inline style beats presentation attributes. `renderChart` **returns a `NoteHighlighter`** (`highlight(i)`/`clear()`) that positions an amber DOM-overlay band — a *sibling* of the SVG so the dark-mode `filter: invert()` doesn't recolor it — over the playing note, scaled to the SVG's live width. Re-renders on `document.fonts.ready`.
- **`player.ts`** — playback via **smplr's `Sequencer`** (one `AudioContext`; instruments cached by id — GM `Soundfont`s plus a `"splendid"` kind for the Salamander `SplendidGrandPiano`). Notes sit on an eighth-note grid (PPQ 480). Exposes a transport — `play(spec)`, `togglePause`, `stop`, `setLoop`, `getState`, and `onStateChange`/`onNote` listeners. `SequenceSpec` carries `octaveShift` (×12 semitones, playback-only), swing (`swing`/`swingRatio`/`accent`/`downbeat`: the off-beat lands at `ratio/(ratio+1)` of the beat with long-short durations, and off-beat velocity = `downbeat + accent`), and `loop`. Loop end is snapped to a whole beat so swung/odd-length loops stay even; a small `lookaheadMs` keeps `noteOn` (which drives the highlight) close to the audio.
- **`sequence.ts`** — builds the play-order note sequence shared by the staff and the player. `INTERVAL_PATTERNS` + `sequenceForChart(chart, id)` turn a chart into a **two-measure, 16-note phrase that loops** (the note after the last is the tonic again): `steps` is the scale (up to the 9th and back); `seconds` is a down-a-2nd zigzag (1 7 2 1 3 2 …); `thirds`…`sevenths` pair each degree with the one a 3rd/…/7th above — diatonic degree shifts *within* the scale (octave-wrapped, spelling preserved from `scales.json` — never chromatic). Labels are the W/H steps (Steps) or scale-degree numbers (the rest).
- **`main.ts`** — builds the controls and wires render + transport. Controls: key, scale, **Sound** (GM instruments grouped by `<optgroup>` — keys/mallets, guitar, bass, horns & winds, plus the Salamander grand piano — each carrying a sensible default **octave** applied on selection), **Octave** (−3…+3 playback transpose), Tempo (BPM, quarter-note), Accidentals (auto/sharps/flats), **Intervals** (diatonic practice patterns — Steps, Seconds, Thirds … Sevenths — each a looping two-measure up-and-down phrase from `sequence.ts`; replaces the old Retrograde toggle), and **Feel** (a swing-ratio dropdown of presets after Iverson — Straight 1:1 … Dotted 3:1 — plus off-beat accent and down-beat level sliders that enable only for a swung feel; Straight plays even eighths with uniform velocity). Transport: **icon** buttons — Play/Pause (▶/⏸, tracks `TransportState`), Stop, Loop. The header carries the **version pill** (`__APP_VERSION__`, injected from `package.json` by `vite.config.ts`'s `define`), the dark/light **theme toggle** (persisted in `localStorage`), an **Offline** button (opens the offline-sounds picker), and an **Install** button (Chromium `beforeinstallprompt`; Safari shows an Add-to-Home-Screen/Dock hint). Changing any audio-affecting control stops playback so the next Play rebuilds the sequence.
- **`offline.ts` + PWA** — `vite-plugin-pwa` (`vite.config.ts`) builds the manifest + icons (from `public/pwa-icon.svg`) and a Workbox service worker that **precaches the app shell** and `CacheFirst`-caches the sound CDNs (`gleitz`, `smpldsnds`, `goldst`) into a **`sound-samples`** cache. `offline.ts` manages that cache for the Offline picker (`download`/`isCached`/`remove`, `storage.estimate()` usage). The shell works offline; sounds are cached per the user's choice. (iOS may evict after disuse — Capacitor is the future "guaranteed offline" path.)

## CI / releases

`.github/workflows/build-book.yml` has three parallel jobs — `scales`, `blues`, `web` — each independent. `scales`/`blues` install `./common` + their subproject; `web` does `npm ci` + typecheck + build. All run on every push/PR. On a `v*` tag, `scales` attaches its assets (book PDF + per-key PDFs/WAVs) to the GitHub Release; **blues is intentionally excluded from releases**. Non-tag runs upload artifacts `jazz-scales-ci`, `jazz-blues-ci`, `jazz-web-ci`. The slide deck is **not** in this workflow — it ships HTML-only via `deploy-web.yml` (no PDF/Release artifact).

`.github/workflows/deploy-web.yml` deploys to **GitHub Pages** (Actions source) on pushes to `main` touching `projects/web` **or `projects/presentation`** (and manual dispatch). It builds the web app **and** the slide deck, placing the deck under `dist/slides`, so the app is live at **https://jazz-scales.gkt.sh/** and the deck at **https://jazz-scales.gkt.sh/slides/** (custom subdomain via `public/CNAME`). `vite.config.ts` sets `base: "/"` for root-relative asset paths.
